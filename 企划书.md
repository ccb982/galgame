
功能

构建一个在线Galgame（视觉小说）播放器网站，用户可以浏览游戏列表，选择游戏并开始游玩。播放器支持文字显示、背景切换、立绘展示、选项分支，并允许存档/读档（基于游客ID）。用户认证功能暂不实现，但数据库预留用户表和相关字段，便于后续扩展。


技术栈

后端：Spring Boot 2.x/3.x + Spring Data JPA + MySQL + Lombok + Maven
- Spring Boot：快速构建Java应用的框架，提供自动配置和内嵌服务器，简化开发流程
- Spring Data JPA：简化数据库操作的ORM框架，通过Repository接口自动生成SQL查询，减少代码量
- MySQL：关系型数据库，用于存储游戏数据、用户数据和存档信息
- Lombok：通过注解减少样板代码，如getter、setter、构造函数等
- Maven：项目管理和依赖管理工具，统一构建流程

前端：React 18 + React Router + Axios + TailwindCSS（或Ant Design）+ Zustand（可选）
- React 18：流行的前端JavaScript库，用于构建用户界面，支持组件化开发和虚拟DOM
- React Router：React的路由库，实现单页应用的页面导航和路由管理
- Axios：基于Promise的HTTP客户端，用于前端与后端API的通信
- TailwindCSS：实用优先的CSS框架，通过原子类快速构建响应式界面
- Zustand：轻量级状态管理库，用于管理应用的全局状态（可选）

部署：阿里云ECS (Ubuntu) + Nginx + OpenJDK 17 + MySQL 8.0
- 阿里云ECS：弹性计算服务，提供稳定的服务器环境
- Nginx：高性能Web服务器，用于反向代理和静态资源托管
- OpenJDK 17：Java运行环境，用于运行Spring Boot应用
- MySQL 8.0：生产环境数据库，存储应用数据

开发流程

一、总体开发原则
迭代式开发：不要试图一次性完成所有功能，而是分阶段逐步实现，每个阶段都有可运行的成果。

先定义接口：前后端分离开发时，最好先约定好 API 接口（路径、方法、参数、返回值），然后前后端可以并行开发。

从简单到复杂：先实现基础功能（如游戏列表、详情），再攻克核心难点（播放器逻辑），最后添加辅助功能（存档、管理后台）。

及时测试：每完成一个小功能，就用 Postman 或前端页面测试，确保正确性。

二、详细开发顺序（分阶段）
阶段1：项目初始化与环境搭建
目标：创建前后端项目骨架，确保能正常运行。

后端：

使用 Spring Initializr 生成 Spring Boot 项目，选择依赖：Spring Web、Spring Data JPA、MySQL Driver、Lombok。
配置 application.yml，连接本地 MySQL 数据库（创建空数据库 galgame）。
编写一个简单的测试 Controller（例如返回 "Hello"），启动项目验证。
前端：

使用 Vite 创建 React 项目：npm create vite@latest galgame-frontend -- --template react。
安装依赖：react-router-dom、axios、tailwindcss（按需配置）。
运行 npm run dev，看到默认页面。
输出：一个能跑起来的后端服务和一个空白的前端页面。

阶段2：数据库设计与实体类
目标：根据需求设计数据库表，创建 JPA 实体和 Repository。

任务：

设计表结构：user（预留）、game、scene、save（参考之前的 SQL）。
在 entity 包下创建对应的实体类，使用 JPA 注解（@Entity、@Id、@GeneratedValue 等），并添加 Lombok 注解（@Data、@NoArgsConstructor）简化代码。
创建 Repository 接口继承 JpaRepository，并添加一些基础查询方法（如 findByGameIdAndSceneKey）。
配置 ddl-auto: update，启动项目让 JPA 自动建表，检查表结构是否正确。
输出：数据库表自动生成，Repository 接口就绪。

阶段3：基础 API 开发与前端对接
目标：实现游戏列表和详情接口，让前端能获取数据并展示。

后端：

创建 GameController，实现：
GET /api/games：分页查询游戏列表（上架状态为 1），支持 page 和 size 参数。
GET /api/games/{id}：根据 ID 查询游戏详情。
创建 GameService 处理业务逻辑（调用 Repository）。
测试 API（用 Postman）。
前端：

创建 API 模块（src/api/game.js），封装 Axios 请求。
创建路由：首页 /（游戏库）、详情页 /game/:id。
首页组件调用 API 获取游戏列表，展示卡片网格（可用 TailwindCSS 快速布局）。
详情页组件根据路由参数调用 API 获取游戏详情并展示。
输出：前端能显示从后端获取的游戏列表和详情，实现基础数据交互。

阶段4：播放器核心（场景加载与渲染）
目标：实现播放器页面，能够加载剧本、渲染背景、立绘、文字，并支持点击推进（无分支时）。

后端：

创建 SceneController，实现 GET /api/games/{gameId}/scenes/{sceneKey}，返回场景的 JSON 剧本。
预先在数据库中添加一些测试数据（手动插入几条场景记录）。
前端：

创建播放器页面 Player，路由 /play/:gameId?scene=start。
从 URL 获取 gameId 和 sceneKey（默认 start），调用场景 API 获取剧本。
解析剧本 JSON，动态渲染：
背景图：<img> 绝对定位。
立绘：根据 characters 数组渲染多张图片，通过 CSS 定位。
文本框：底部半透明黑底白字，显示 text。
处理点击事件：如果剧本中有 next_scene，则加载下一场景；如果有 options，则显示选项按钮（先不处理分支）。
实现简单的场景推进（无选项时点击屏幕继续）。
输出：播放器能加载并显示第一个场景，点击后能进入下一个场景（如果有预设的 next_scene）。

阶段5：选项分支与变量系统
目标：支持多选项分支，并记录简单变量（如好感度）。

后端：

修改场景表，variables 字段用于存储变量（可选，暂不实现复杂变量逻辑）。
场景 JSON 中添加 options 数组，每个选项包含 text 和 next。
前端：

当剧本中包含 options 时，隐藏文本框点击，显示选项按钮。
点击选项后，根据 next 跳转到对应场景。
（可选）实现变量系统：在 Zustand store 中维护一个 variables 对象，选项可以修改变量，后续场景根据变量显示不同文本（需要剧本设计支持）。
输出：播放器能处理分支剧情，用户选择不同选项进入不同场景。

阶段6：存档/读档功能
目标：实现基于游客 ID 的存档和读档。

后端：

创建 SaveController，实现：
GET /api/saves?gameId={gameId}&visitorId={visitorId}：获取某游戏下的所有存档。
POST /api/saves：创建存档（传入 gameId, sceneKey, variables, visitorId）。
PUT /api/saves/{id}：更新存档（覆盖）。
DELETE /api/saves/{id}：删除存档。
在 Save 实体中，user_id 字段暂时存放游客 ID（可视为游客标识）。
前端：

生成游客 ID：应用启动时检查 localStorage，若无则生成 UUID 并存入 Zustand store 和 localStorage。
在播放器页面添加“保存”和“读取”按钮：
点击保存：调用存档 API，传入当前场景和变量，成功后刷新存档列表。
点击读取：弹出存档列表，选择后加载对应场景。
在游戏详情页，可显示该游戏的存档列表（可选）。
输出：用户可以在游戏中保存进度，之后能读取继续。

阶段7：游戏管理后台（可选）
目标：为了方便添加游戏和剧本，开发一个简单的管理界面（如果愿意）。

后端：

添加上传接口：POST /api/admin/games 用于新增游戏（可包含封面上传），POST /api/admin/scenes 用于批量导入剧本 JSON。
使用 MultipartFile 处理文件上传。
前端：

创建简单的管理页面（可放在 /admin 路由下），需要简单鉴权（可以写死一个管理员账号，或先不做权限）。
实现表单上传游戏信息，以及上传剧本 JSON 文件。
输出：可以通过管理页面向数据库添加新游戏，而不必手动操作数据库。

阶段8：测试、优化与部署
目标：整体测试，修复 Bug，优化体验，然后部署到阿里云。

任务：

功能测试：遍历所有流程，确保存档、读档、分支正常。
性能优化：图片懒加载、代码分割（React.lazy）、后端索引优化。
错误处理：前后端添加统一错误提示。
部署准备：
后端修改配置文件为生产环境（application-prod.yml），设置 ddl-auto=validate 或 none。
前端构建：npm run build，生成静态文件。
部署：
将后端打包成 jar，上传到服务器运行。
将前端构建后的文件上传到服务器，用 Nginx 托管，并配置反向代理 API。
配置 MySQL，导入表结构。
测试线上环境。
输出：一个可公开访问的 Galgame 播放器网站。

三、前后端协作建议
接口文档：可以使用 Swagger（Spring Boot 集成 springdoc-openapi）自动生成 API 文档，或者简单地在代码注释中写明，甚至用 Postman 分享接口示例。

跨域问题：开发时前端代理解决（Vite 配置 proxy），生产环境由 Nginx 反向代理解决。

数据格式：统一使用 JSON，日期时间用 ISO 8601 格式。

四、注意事项
剧本 JSON 格式：初期可以简化，例如：

json
{
  "bg": "bg_school.jpg",
  "characters": [{"name": "alice", "image": "alice_happy.png", "position": "left"}],
  "text": "今天天气真好。",
  "next": "scene2"
}
有选项时用 options 数组代替 next。

图片存储：开发时图片可放在后端 static 目录下，方便访问；生产环境建议用对象存储（如阿里云 OSS）或 Nginx 直接托管。

游客 ID：前端生成后保存在 localStorage，每次请求附加在请求头（如 X-Visitor-Id）或作为参数传递。后端从请求中获取并存入存档表。

事务：存档操作涉及数据库写入，用 @Transactional 确保原子性。

分页：游戏列表用 Spring Data 的 Page 对象返回，前端可获取总页数等信息。
